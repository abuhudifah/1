
# نظام أبو حذيفة المتكامل للصرافة والتحويلات  
**Abu Hudhaifa Integrated Exchange & Transfer System**  
نظام مالي يعمل بـ **Online-First**، حيث Supabase هو المصدر الوحيد الموثوق للبيانات. الذاكرة المحلية (IndexedDB) مخصصة فقط لاستمرارية العمل أثناء انقطاع الإنترنت المفاجئ.

---

## 🚀 التشغيل السريع

```bash
git clone [رابط المستودع]
cd abu-hudhaifa
cp config.example.js config.js   # ثم أضف بيانات Supabase الخاصة بك
npx serve .                       # أو python -m http.server 8080
```

ملاحظة: النظام يعتمد على اتصال إنترنت للقراءة والكتابة العادية. دعم وضع الطوارئ (بدون إنترنت) متاح فقط للمستخدمين الذين سبق لهم تسجيل الدخول على نفس الجهاز.

🧠 السلوك المعماري (Online-First)

الحالة مصدر القراءة مصدر الكتابة دور IndexedDB
إنترنت متوفر Supabase مباشرة Supabase مباشرة فقط تحديث خلفي لنسخة طوارئ (لا يُستخدم في التدفق الرئيسي)
إنترنت متوفر لكن Dexie تالفة Supabase مباشرة Supabase مباشرة يحاول إعادة الفتح في الخلفية – لا تأثير على الوظائف
انقطاع الإنترنت (طوارئ) Dexie (للبيانات المخزنة مسبقاً) Dexie + SyncQueue هنا فقط يُستخدم Dexie لتخزين العمليات الجديدة مؤقتاً
عودة الإنترنت بعد الطوارئ Supabase مباشرة Supabase + المزامنة تتم معالجة SyncQueue ورفع العمليات إلى Supabase ثم حذفها محلياً

🔐 تسجيل الدخول

· لا يعتمد على IndexedDB أبداً (إلا في حالة الدخول السريع أثناء انقطاع الإنترنت وعلى نفس الجهاز).
· الدخول السريع بالمعادلة يعمل عبر SHA-256 → RPC في Supabase (عند وجود اتصال) أو البحث في Dexie (في وضع الطوارئ).

⏱️ إدارة الخمول (IdleTimer)

· المندوبون يخضعون لمؤقت خمول مدته 5 دقائق – عند انتهائه يتم تسجيل الخروج تلقائياً.
· المدير والمساعد الإداري لا يخضعان للمؤقت (يبقيان مسجلين حتى تسجيل الخروج اليدوي).
· يتم إعادة تعيين المؤقت عند كل تفاعل للمستخدم (نقر، كتابة، تحريك الماوس).

🔄 المزامنة الاحترافية

· تعمل في الخلفية كل 30 ثانية (أو عند إضافة عمليات جديدة).
· إعادة محاولة بتأخير تصاعدي + Jitter (حتى 5 محاولات، ثم تنقل إلى sync_conflicts للمراجعة اليدوية).
· الكشف عن التعارضات عبر مقارنة updated_at.
· المعرفات المؤقتة (TEMP_...) تُستبدل بالمعرفات الحقيقية من Supabase بعد المزامنة الناجحة (باستخدام معاملة ذرية في Dexie).
· تنظيف دوري: حذف المعاملات المزامنة الأقدم من 90 يوماً، وعمليات الانتظار الأقدم من 30 يوماً.

🚪 تسجيل الخروج

· عند الخروج: إلغاء اشتراكات Realtime، مسح sessionStorage، إيقاف مؤقتات المزامنة و IdleTimer.
· لا تُمسح Dexie (تبقى لاستخدام الطوارئ لاحقاً).

🧹 إعادة ضبط البيانات المحلية (للمدير فقط)

· زر في الإعدادات يظهر لدور admin.
· يحذف جميع جداول Dexie، ويمسح localStorage و sessionStorage، ثم يعيد تحميل الصفحة.
· لا يؤثر على Supabase ولا على أجهزة المستخدمين الآخرين.

---

🧪 السيناريوهات الحرجة المدعومة

السيناريو السلوك المتوقع
المستخدم يحذف Dexie يدوياً من Console أثناء تشغيل التطبيق لا انهيار؛ يتم إعادة فتح Dexie في الخلفية؛ تسجيل الدخول والعملات تبقى عاملة عبر Supabase مباشرة.
انقطاع الإنترنت أثناء إدخال عملية تُحفظ العملية في Dexie وتُضاف إلى sync_queue. رسالة "تم الحفظ محلياً...".
عودة الإنترنت قبل تسجيل الخروج تُشغّل المزامنة تلقائياً وترفع العمليات إلى Supabase دون الحاجة لإعادة تسجيل الدخول.
نفس المستخدم يدخل من جهازين في وضع الطوارئ يحتفظ كل جهاز بعملياته المحلية الخاصة. عند اتصالهما، تظهر تعارضات في sync_conflicts للمدير.
المندوب يترك الجهاز دون تفاعل لمدة 5 دقائق يتم تسجيل خروجه تلقائياً (بسبب IdleTimer).
المدير يضغط "إعادة ضبط البيانات المحلية" تُمسح Dexie محلياً فقط على جهازه؛ لا تأثير على أجهزة الآخرين ولا على Supabase.
