## 1. ملخص طبيعة العمل

### 1.1 طبيعة النشاط التجاري

نظام أبو حذيفة هو نظام إدارة عمليات **شركة صرافة وتحويلات** تعمل بالأسلوب التالي:
- مناديب ميدانيون يتعاملون نقدياً مع العملاء في الميدان
- كل مندوب يحمل **عهدة نقدية** (صندوق شخصي) يتحمل مسؤوليتها
- يودعون ما يجمعونه في **حسابات بنكية** تابعة لـ**شركات** تعمل معها المؤسسة
- الإدارة تراقب الحركات وتُقفل يومياً وتُسوّي الحسابات

### 1.2 الأطراف الرئيسية

| الطرف | دوره في العمل | تمثيله في النظام |
|-------|---------------|------------------|
| **المندوب** | يُجمع ويودع ويُسلّم النقد ميدانياً | `users` بـ role='agent' + حساب `AGT_<id>` |
| **الشركة** | جهة الإيداع — تستقبل النقد في بنوكها | جدول `companies` + حساب `COMP_<id>` |
| **الحساب البنكي** | وعاء استقبال الإيداعات — تابع لشركة | جدول `bank_accounts` + حساب `BNK_<id>` |
| **العميل / المدين** | يدفع ديوناً أو أموالاً للمندوب | جدول `debtors` + حساب `CUST_<id>` |
| **الإدارة** | تُشرف وتُقفل وتُصحح | `users` بـ role='admin'/'admin_assistant' |

### 1.3 المبادئ التشغيلية الجوهرية

1. **كل حركة نقدية تُنشئ قيداً محاسبياً فورياً** — لا حركة بدون قيد.
2. **كل قيد يؤثر على حسابَين بالضبط** — مدين ودائن، مجموعهما متساوٍ.
3. **عهدة المندوب تتحرك فقط بثلاثة أسباب:** استلام نقد (ترتفع)، إيداع/تسليم (تنخفض)، مصروف (تنخفض).
4. **الحساب البنكي للشركة يتحرك فقط عند الإيداع** — لا حركة أخرى مباشرة عليه.
5. **حساب الشركة الأم** يعكس مجموع حسابات بنوكها.
6. **أي عملية بين طرفين تُقيَّد على الطرفين** — عدم الاعتراف بالعملية أحادياً.

## 2. خريطة العمليات التشغيلية

### 2.1 عمليات المندوب اليومية

```
┌─────────────────────────────────────────────────────────────────────┐
│                       عمليات المندوب اليومية                        │
│                                                                     │
│  ① تحصيل من شركة          النقد يدخل الصندوق من ذمة الشركة        │
│  ② تحصيل من عميل مدين     النقد يدخل الصندوق ويُخفَّض الدين       │
│  ③ سحب من بطاقة بنكية     النقد يُسحب من بنك → يدخل الصندوق      │
│  ④ إيداع بنكي             النقد يخرج من الصندوق → يدخل البنك      │
│  ⑤ مصروف تشغيلي           النقد يخرج من الصندوق → مصروف           │
│  ⑥ استلام من مندوب آخر    النقد يدخل صندوقه من صندوق الآخر        │
│  ⑦ تسليم لمندوب آخر       النقد يخرج من صندوقه → صندوق الآخر      │
│  ⑧ استلام من شركة         النقد يدخل الصندوق من ذمة الشركة        │
│  ⑨ تسليم لشركة            النقد يخرج من الصندوق → ذمة الشركة      │
└─────────────────────────────────────────────────────────────────────┘
```

### 2.2 عمليات إدارية

```
┌─────────────────────────────────────────────────────────────────────┐
│                       العمليات الإدارية                             │
│                                                                     │
│  ⑩ تسوية إيداع فاشل       إلغاء إيداع → استرداد المبلغ للمندوب   │
│  ⑪ عكس معاملة             إلغاء قيد خاطئ بقيد عكسي               │
│  ⑫ إقفال يومي             تجميد الأرصدة اليومية كـ snapshot        │
│  ⑬ إضافة مدين جديد        تسجيل عميل مديون جديد                  │
│  ⑭ قيد يدوي               تسجيل حركة استثنائية بإذن إداري         │
└─────────────────────────────────────────────────────────────────────┘
```

### 2.3 دورة يوم عمل كاملة

```
صباحاً:
  المندوب يُحدد رصيد افتتاح صندوقه ← يجب مطابقة النظام

خلال اليوم:
  ← تحصيلات من شركات / مدينين / عملاء
  ← إيداعات في بنوك الشركات
  ← مصروفات عارضة
  ← استلام وتسليم بين المناديب

مساءً:
  المندوب يُسلّم/يودع كل ما تجمّع
  الإدارة تُراجع الملخص اليومي
  الإدارة تُقفل اليوم ← snapshot لجميع الأرصدة
```

## 3. خريطة العمليات المحاسبية

### 3.1 تدفق النقد ومساراته

```
                        ┌────────────────────┐
                        │   GENERAL_FUND     │
                        │ (الصندوق المركزي)  │
                        └────────┬───────────┘
                                 │ مصدر عام (بدون جهة محددة)
                     ┌───────────┼───────────┐
                     ↓           ↓           ↓
            ┌──────────────┐  ┌──────────────┐  ┌──────────────┐
            │  AGT_<id>    │  │  COMP_<id>   │  │  CUST_<id>   │
            │ صندوق مندوب │  │ حساب شركة   │  │ حساب مدين   │
            └──────┬───────┘  └──────┬───────┘  └──────┬───────┘
                   │                  │                  │
          إيداع ↓  │ ↑ تحصيل         │                  │ ↑ تحصيل من مدين
                   │                  │
            ┌──────────────┐
            │  BNK_<id>    │
            │ حساب بنكي   │
            └──────────────┘
                   ↑
              مرتبط بـ COMP_ (الشركة المالكة)
```

### 3.2 الأثر المحاسبي لكل عملية

| العملية | حساب يرتفع (DR) | حساب ينخفض (CR) |
|---------|----------------|----------------|
| تحصيل من شركة | AGT_ (صندوق المندوب) | COMP_ (ذمة الشركة) |
| تحصيل من مدين | AGT_ (صندوق المندوب) | CUST_ (ذمة المدين تنخفض = سداد) |
| سحب من بطاقة | AGT_ (صندوق المندوب) | BNK_ (رصيد البنك ينخفض) |
| إيداع في بنك | BNK_ (رصيد البنك يرتفع) + يُسوّى COMP_ | AGT_ (صندوق المندوب) |
| مصروف تشغيلي | EXP_ (سجل المصروف) | AGT_ (صندوق المندوب) |
| استلام من مندوب | AGT_receiver (يرتفع) | AGT_sender (ينخفض) |
| تسليم لمندوب | AGT_receiver (يرتفع) | AGT_sender (ينخفض) |
| استلام من شركة | AGT_ (صندوق المندوب) | COMP_ (ذمة الشركة) |
| تسليم لشركة | COMP_ (ذمة الشركة ترتفع) | AGT_ (صندوق المندوب) |
| تسوية إيداع فاشل | AGT_ (يعود المبلغ) | COMP_ (تُعكس ذمتها) |

## 4. تعريف الحسابات الأساسية

### 4.1 صندوق المندوب — AGT_

```
المعرّف     : AGT_<user_uuid>
الطبيعة     : أصل (Asset)
يمثّل       : النقد الفعلي في حوزة المندوب
يرتفع عند  : تحصيل، استلام، سحب من بطاقة
ينخفض عند  : إيداع، تسليم، مصروف
الرصيد الطبيعي: مدين (Debit balance)
يملكه      : المندوب شخصياً — مسؤوليته الكاملة
```

### 4.2 حساب الشركة — COMP_

```
المعرّف     : COMP_<company_uuid>
الطبيعة     : التزام تجاه الشركة (Liability)
يمثّل       : ما يستحقه النظام للشركة (أو ما للنظام عليها)
يرتفع (CR) عند: تحصيل لصالح الشركة، إيداع بنكي، تسليم نقدي للشركة
ينخفض (DR) عند: استلام من الشركة، تسوية إيداع فاشل
الرصيد الطبيعي: دائن (Credit balance) — النظام مدين للشركة
ملاحظة     : هذا الحساب يجمع كل العلاقة المالية مع الشركة
```

### 4.3 الحساب البنكي — BNK_

```
المعرّف     : BNK_<bank_account_uuid>
الطبيعة     : أصل (Asset)
يمثّل       : الرصيد المُودَع في هذا الحساب البنكي
يرتفع عند  : إيداع بنكي (المندوب يودع)
ينخفض عند  : سحب من بطاقة (عميل يسحب من البطاقة المرتبطة)
الرصيد الطبيعي: مدين (Debit balance)
تبعيته     : تابع لـ COMP_ (كل بنك يخص شركة)
العلاقة    : مجموع BNK_ التابعة لشركة = ناقص المسحوبات
```

### 4.4 حساب المدين — CUST_

```
المعرّف     : CUST_<debtor_uuid>
الطبيعة     : أصل (Asset) — مستحقات على العميل
يمثّل       : المبلغ المتبقي الذي يدين به العميل
يرتفع (DR) عند: تسجيل دين جديد على العميل
ينخفض (CR) عند: تحصيل من العميل (سداد جزئي أو كامل)
الرصيد الطبيعي: مدين (Debit balance) — للنظام على العميل
```

**تصحيح جوهري:** في الكود الحالي القيد مقلوب — يجب تصحيحه:
```
عند التحصيل من مدين:
صحيح:   DR AGT_   (الصندوق يرتفع)    CR CUST_  (الدين ينخفض)
خاطئ:   DR CUST_  (الدين يرتفع!)     CR AGT_   (الصندوق ينخفض!)
```

### 4.5 حساب المصروف — EXP_

```
المعرّف     : EXP_<expense_code>
الطبيعة     : مصروف (Expense)
يمثّل       : نوع المصروف التشغيلي
يرتفع عند  : صرف مبلغ من صندوق المندوب
الرصيد الطبيعي: مدين (Debit balance) — يتراكم خلال الفترة
```

**قاعدة صارمة:** expense_code يجب أن يكون الرمز النظيف فقط (مثل `FUEL`, `TRANSPORT`) — لا يبدأ بـ `EXP_` لأن البادئة تُضاف تلقائياً.

### 4.6 الصندوق المركزي — GENERAL_FUND

```
المعرّف     : GENERAL_FUND (ثابت)
الطبيعة     : حساب التوازن العام
يمثّل       : الطرف الثاني في العمليات التي لا يُحدَّد فيها مصدر أو وجهة
يُستخدم عند: تحصيل بدون شركة أو عميل، استلام/تسليم بدون مصدر محدد
```

### 4.7 حساب الموقوف — SUSP_ (مُعرَّف لكن غير مُفعَّل)

```
المعرّف     : SUSP_<reference>
الطبيعة     : حساب مؤقت (Suspense)
يمثّل       : معاملات قيد المعالجة أو الانتظار
متى يُستخدم: إيداع بنكي قيد التأكيد، تحويل بين مناديب لم يُستلم بعد
```

## 5. قواعد الاستلام والتسليم

### 5.1 المشكلة الحالية في الواجهة

الواجهة الحالية (`DataEntryComponent.js` — نموذج الاستلام/التسليم) تعتمد على:
- قائمة منسدلة لاختيار نوع الجهة (مندوب / شركة)
- بحث نصي في أسماء المناديب
- حقل نصي اختياري لرقم الحساب

**المشكلة:** لا يوجد تحقق من رقم الحساب، والنظام يعتمد على الاسم لا على المعرّف.

### 5.2 النموذج التشغيلي المطلوب

```
الاستلام والتسليم يجب أن يعمل بالأسلوب التالي:

الخطوة 1: المستخدم يُدخل رقم الحساب
         ← رقم موحَّد لكل كيان في النظام (مندوب، شركة، حساب بنكي)

الخطوة 2: النظام يبحث عن الحساب فوراً
         ← إذا وُجد: يعرض اسمه وطبيعته (مندوب / شركة)
         ← إذا لم يُوجد: يطلب من المستخدم تأكيد الاسم ويحفظه

الخطوة 3: بعد أول استخدام، يُحفظ الحساب كمستفيد محفوظ
         ← يظهر في قائمة "الجهات المعتادة" بدون إعادة إدخال

الخطوة 4: المستخدم يُدخل المبلغ ويُؤكد
```

### 5.3 قواعد رقم الحساب

```
الصيغة    : <نوع>-<معرّف>
مثال مندوب: AGT-001   أو   AGT-<اسم مختصر>
مثال شركة : COMP-001  أو   رقم السجل التجاري
مثال بنك  : BNK-001   أو   آخر 4 أرقام من رقم الحساب

قاعدة التحقق:
- إذا كان الحساب موجوداً في النظام → يُنفَّذ مباشرة
- إذا كان خارجياً (غير مُسجَّل) → يُدخل الاسم ويُحفظ كـ "جهة خارجية"
- أول استخدام لجهة خارجية → يُطلب التأكيد الإداري (اختياري)
```

### 5.4 تعارض المزدوج: RECEIPT + DELIVERY لنفس العملية

```
المشكلة: إذا سجّل مندوب A "تسليم" لـ B، وسجّل B "استلام" من A،
          يُسجَّل القيد مرتين في account_ledger.

الحل التشغيلي المطلوب:
إما أن يسجّل طرف واحد (A يُسلّم → يُسجَّل DELIVERY فقط)
وينعكس تلقائياً على B،
أو: نظام موافقة — A يُبدئ → B يُوافق → قيد واحد.

السلوك الحالي: كل طرف يُسجَّل منفصلاً → مخاطر تكرار.
```

## 6. قواعد الإيداع البنكي

### 6.1 طبيعة عملية الإيداع البنكي

```
العملية: المندوب يأخذ نقداً من صندوقه → يودعه في بنك تابع لشركة

الأثر الاقتصادي:
  صندوق المندوب: ينخفض بمقدار المبلغ
  الحساب البنكي: يرتفع بمقدار المبلغ
  حساب الشركة  : يُعتبر وسيطاً مؤقتاً (تُسوَّى تلقائياً)
```

### 6.2 هيكل القيدين الحالي (مراجعة تقنية)

```
القيد الحالي (قسيمتان):

قسيمة 2:
  DR  BNK_<bank>       amount   ← البنك يرتفع ✅
  CR  COMP_<company>   amount   ← ذمة الشركة تُستكمل

قسيمة 3:
  DR  COMP_<company>   amount   ← تُصفَّر ذمة الشركة
  CR  AGT_<agent>      amount   ← صندوق المندوب ينخفض ✅

الصافي:
  AGT_  : -amount  ✅ (الصندوق ينخفض)
  COMP_ : 0        (يُصفَّر — لا أثر صافٍ على ذمة الشركة)
  BNK_  : +amount  ✅ (البنك يرتفع بشكل دائم)
```

**ملاحظة:** التصميم الحالي يُصفِّر أثر COMP_ في كل إيداع، مما يعني أن حساب الشركة لا يسجّل تراكماً للإيداعات. هذا صحيح من ناحية الإيداع، لكن يُفقد معلومة "كم أودع المندوب لصالح هذه الشركة".

### 6.3 القاعدة التشغيلية للإيداع

```
قاعدة 1: الإيداع يتطلب دائماً تحديد الحساب البنكي (bank_account_id)
قاعدة 2: الحساب البنكي مرتبط دائماً بشركة (company_id) — علاقة إلزامية
قاعدة 3: السقف اليومي يُحتسب على مستوى الحساب البنكي لا الشركة
قاعدة 4: عند تجاوز السقف → تنبيه وطلب تأكيد، ليس رفضاً تلقائياً
قاعدة 5: إيداع ناجح → يُحدَّث رصيد AGT_ فوراً (offline أو online)
```

### 6.4 سحب من بطاقة بنكية

```
هذه عملية مختلفة عن الإيداع:
  المندوب يسحب نقداً من بطاقة بنكية بدلاً من استلامه نقداً

القيد:
  DR  AGT_<agent>      amount   ← صندوق المندوب يرتفع
  CR  BNK_<bank>       amount   ← رصيد البنك ينخفض

ملاحظة: هذا يُعكس تأثير الإيداع تماماً.
التسمية الحالية: "تحصيل — سحب من بطاقة" ← مضللة
التسمية الصحيحة: "سحب بنكي" أو "استلام بنكي"
```

## 7. قواعد التحصيل

### 7.1 التحصيل من شركة

```
الوصف    : المندوب يستلم نقداً من عميل لصالح شركة
           (يجمع أقساطاً، مدفوعات، دفعات)
صاحب العملية: المندوب
الطرف الآخر : الشركة (الجهة التي يُجمع لصالحها)
الأثر     :
  DR AGT_<agent>       ← صندوق المندوب يرتفع
  CR COMP_<company>    ← ذمة الشركة على النظام ترتفع
الاكتمال  : فور تسجيل المبلغ
لا تعليق  : هذه عملية فورية — لا تحتاج موافقة
```

### 7.2 التحصيل من مدين (سداد دين)

```
الوصف    : عميل مديون يسدّد ما عليه للمندوب
صاحب العملية: المندوب
الطرف الآخر : العميل المدين (debtor)
الأثر     :
  DR AGT_<agent>       ← صندوق المندوب يرتفع (استلام نقد)
  CR CUST_<debtor>     ← ذمة المدين تنخفض (سداد الدين)
الأثر الجانبي: يُحدَّث سجل المديونية في جدول debtors
الاكتمال  : فور التسجيل + تحديث رصيد المديونية
```

**تصحيح جوهري للكود:** الكود الحالي يعكس القيد خطأً (CR AGT_ / DR CUST_) — يجب تصحيحه.

### 7.3 تحصيل عام (بدون تحديد جهة)

```
الوصف    : تحصيل نقدي لا ينتسب لشركة أو عميل محدد
الأثر     :
  DR AGT_<agent>       ← صندوق المندوب يرتفع
  CR GENERAL_FUND      ← الصندوق المركزي للتوازن
ملاحظة   : هذه الحالة يجب تجنبها قدر الإمكان لأنها تفقد المعلومة
```

### 7.4 سداد المديونية — قواعد خاصة

```
قاعدة 1: عند تسديد دين، يجب تحديد الـ debtor_id
قاعدة 2: المبلغ المُحصَّل ≤ المبلغ المتبقي في المديونية
          → إذا تجاوز: تنبيه للمراجعة (قد يكون دفعة زائدة)
قاعدة 3: عند السداد الكامل: debt_amount → 0 في جدول debtors
قاعدة 4: يُسجَّل تاريخ ومبلغ كل دفعة في سجل منفصل
```

## 8. قواعد الحسابات البنكية للشركات

### 8.1 الهيكل الهرمي المطلوب

```
الشركة (COMP_<id>)
    │
    ├── الحساب البنكي أ (BNK_<id1>)  ← إيداعات مندوب 1
    ├── الحساب البنكي ب (BNK_<id2>)  ← إيداعات مندوب 2
    └── الحساب البنكي ج (BNK_<id3>)  ← إيداعات مشتركة
```

### 8.2 القاعدة: الحساب البنكي يرث الشركة

```
كل حساب بنكي (bank_account) له:
  company_id: (إلزامي) — ربط صريح بالشركة

عند أي حركة على BNK_:
  → يُعرَض على شاشة الشركة الأم ضمن تفاصيل حساباتها البنكية
  → يُستخدم في حساب الإجمالي اليومي للشركة
```

### 8.3 ما يُعرَض في تبويب الحسابات البنكية

**الوضع الحالي:** كل بطاقة بنك تعرض:
- اسم البنك وسقفه
- إجمالي إيداعات اليوم وشريط تقدم
- آخر 5 إيداعات

**الوضع المطلوب إضافياً:**
- تجميع البنوك تحت كل شركة
- عرض إجمالي الشركة = مجموع بنوكها
- تفاصيل إيداعات كل بنك بالتواريخ والمناديب

### 8.4 قاعدة الحساب الأم للشركة

```
إجمالي الشركة = Σ (إجمالي إيداعات كل BNK_ تابع لها)
رصيد COMP_<id> = ما حصلته الشركة حتى الآن من المناديب

ملاحظة هامة:
في التصميم الحالي، COMP_ يُصفَّر في كل إيداع (بسبب القسيمتين).
هذا يعني COMP_ لا يتراكم → لا يمكن قراءة "إجمالي ذمة الشركة" منه.
الحل: إضافة جدول تجميعي أو إعادة تصميم قيد الإيداع.
```

## 13. Business Rules

### قواعد التحصيل

**BR-001**
- If: المندوب يُسجّل تحصيلاً مع `customer_id`
- Then: القيد هو DR AGT_ / CR CUST_، ويُحدَّث `debt_amount` في جدول `debtors`
- Because: التحصيل من مدين يُقلّل ذمته ويُدخل نقداً لصندوق المندوب

**BR-002**
- If: المبلغ المُحصَّل > المديونية المتبقية
- Then: تنبيه للمراجعة — "المبلغ يتجاوز الدين المسجَّل"
- Because: الزيادة قد تكون دفعة مقدّمة أو خطأ في المبلغ

**BR-003**
- If: المندوب يُسجّل تحصيلاً مع `company_id`
- Then: القيد هو DR AGT_ / CR COMP_
- Because: المندوب يُجمع لصالح الشركة — الشركة تستحق هذا المبلغ

---

### قواعد الإيداع

**BR-010**
- If: المندوب يُسجّل إيداعاً في حساب بنكي
- Then: يجب أن يكون `bank_account_id` مُحدَّداً، ويجب أن يكون البنك مرتبطاً بـ `company_id`
- Because: لا إيداع بدون تحديد وجهته البنكية والجهة المُستفيدة

**BR-011**
- If: الإيداع اليومي للحساب البنكي سيتجاوز السقف
- Then: تنبيه وطلب تأكيد — لا رفض تلقائي
- Because: السقف تحذيري تشغيلياً، قد تكون هناك ظروف استثنائية

**BR-012**
- If: إيداع سُجِّل في الحساب البنكي ثم فشل في البنك
- Then: يُسجَّل كـ `failed_deposit` بحالة 'pending'، ويبقى قيده قائماً حتى التسوية
- Because: الفشل البنكي لا يعني إلغاء التسجيل — يحتاج تسوية رسمية

---

### قواعد الاستلام والتسليم

**BR-020**
- If: مندوب A يُسجّل تسليماً لمندوب B (DELIVERY)
- Then: القيد هو DR AGT_B / CR AGT_A
- Because: النقد انتقل من صندوق A إلى صندوق B

**BR-021**
- If: مندوب B يُسجّل استلاماً من مندوب A (RECEIPT) لنفس المبلغ والتاريخ
- Then: تنبيه "ربما يوجد قيد مكرر — تحقق"
- Because: A سبق أن سجّل التسليم → القيد موجود

**BR-022**
- If: الطرف الآخر في الاستلام/التسليم غير مُسجَّل في النظام
- Then: يُدخل اسمه ويُحفظ كـ "جهة خارجية" — العملية لا تُمنع
- Because: التعامل مع جهات خارجية واقع تشغيلي

---

### قواعد المصروفات

**BR-030**
- If: المندوب يُسجّل مصروفاً
- Then: يجب تحديد نوع المصروف (expense_code) من قائمة المعتمدة
- Because: المصروف العشوائي يُفسد تقارير التكاليف

**BR-031**
- If: expense_code غير موجود في القائمة
- Then: يمكن إضافة نوع جديد، لكن كوده يُولَّد من الاسم بدون بادئة EXP_
- Because: البادئة تُضاف تلقائياً عند بناء معرّف الحساب

---

### قواعد المزامنة المحاسبية

**BR-040**
- If: عملية أُنشئت offline بـ temp_id
- Then: عند المزامنة، يجب تحديث `reference_id` في قيود account_ledger بالـ real_id
- Because: قيود account_ledger تُشير لـ transactions.id عبر FK

**BR-041**
- If: عملية offline اكتملت المزامنة
- Then: يتحول `sync_status` من 'pending' إلى 'synced' في transactions + account_ledger
- Because: الحالة 'synced' تعني: الكيان موجود في Supabase ومطابق

## 15. Naming Convention

### 15.1 معرّفات الحسابات المحاسبية

| النوع | البادئة | صيغة المعرّف | مثال |
|-------|---------|-------------|------|
| صندوق مندوب | `AGT_` | `AGT_<user_uuid>` | `AGT_a1b2c3d4-...` |
| حساب شركة | `COMP_` | `COMP_<company_uuid>` | `COMP_f1e2d3c4-...` |
| حساب بنكي | `BNK_` | `BNK_<bank_account_uuid>` | `BNK_b9a8c7d6-...` |
| حساب مدين | `CUST_` | `CUST_<debtor_uuid>` | `CUST_1234abcd-...` |
| حساب مصروف | `EXP_` | `EXP_<CODE>` | `EXP_FUEL` |
| حساب موقوف | `SUSP_` | `SUSP_<reference>` | `SUSP_DEP_abc123` |
| صندوق مركزي | — | `GENERAL_FUND` | `GENERAL_FUND` |

**قاعدة صارمة:** expense_code يجب أن يُخزَّن بدون بادئة EXP_ في جدول expense_accounts — البادئة تُضاف عند بناء AccountId.expense(code) فقط.

### 15.2 أنواع العمليات — التسمية الموحّدة

| قيمة في الكود | التسمية التشغيلية العربية | الوصف المختصر |
|--------------|--------------------------|---------------|
| `collection` | تحصيل | استلام نقد من شركة أو عميل |
| `bank_withdrawal` | سحب بنكي | استلام نقد من حساب بنكي (جديد) |
| `deposit` | إيداع بنكي | إيداع النقد في حساب بنكي |
| `expense` | مصروف | صرف من الصندوق لمصروف تشغيلي |
| `receipt` | استلام | استلام نقد من مندوب أو شركة |
| `delivery` | تسليم | تسليم نقد لمندوب أو شركة |
| `refund_settlement` | تسوية إيداع فاشل | استرداد مبلغ إيداع فاشل |

### 15.3 حالات العمليات

| الحالة | المعنى التشغيلي |
|--------|----------------|
| `synced` | العملية محفوظة في Supabase وقيودها مُسجَّلة |
| `pending` | العملية محفوظة محلياً وتنتظر المزامنة |
| `conflict` | تعارض في المزامنة — يحتاج مراجعة يدوية |

### 15.4 حالات الإيداع الفاشل

| الحالة | المعنى التشغيلي |
|--------|----------------|
| `pending` | الإيداع فشل وينتظر التسوية |
| `claimed` | الإيداع قيد المطالبة مع الجهة البنكية |
| `refunded` | تمت التسوية باسترداد المبلغ |
| `rejected` | رُفض الاسترداد — يحتاج معالجة استثنائية |
